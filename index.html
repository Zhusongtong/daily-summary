代码是产出，不是投入。大自然在自我修复。 很长一段时间以来，我们把代码当作投入品来对待——美化它、格式化它、装饰它、为它痴迷。我们用复杂的 GUI 来写代码：IDE。语法高亮、tree-sit、代码小地图。快捷键、即时补全、幽灵文字。「你用的是什么配色方案？」 我们熬夜争论 API 和函数体的理想长度。这个 API 的样子够不够好看，别人能读懂吗？ 现在我们把注意力转向真正的投入品：需求、规格、反馈、设计灵感。至关重要的是：生产环境中的投入。我们的 Coding Agent 需要理解用户是如何使用你的应用的，他们遇到了什么错误，然后把这些变成代码。 我们迟早不会再把代码和写代码的人神化。我合作过的最优秀的工程师，始终把代码看作达成目标的手段——一种注定很快会被再次转换的产出。

2026 年工程团队结构的新模式：只需要 2 个人。一个海盗，一个建筑师。 海盗的职责是用最快速度开发有价值的产品功能——通过 vibe coding 来交付。 建筑师的职责是把海盗发现的产品方向转化为可靠、结构化的系统——同样通过 vibe coding，但节奏更慢、逻辑更严密。 每个产品都需要一个海盗，但大多数产品在找到 PMF 之前不需要建筑师——即使需要，也不需要全职。建筑师可以跨多个代码库工作，解决有趣的技术挑战。海盗则在某个产品上全情投入、端到端负责。
